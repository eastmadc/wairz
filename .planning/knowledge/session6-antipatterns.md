# Anti-patterns: Session 6 — Firmware Selectors, Findings firmware_id

> Extracted: 2026-04-03
> Source: session 6 uncommitted changes

## Failed Patterns

### 1. Alembic autogenerate on a drifted schema
- **What was done:** Ran `alembic revision --autogenerate` to create the findings firmware_id migration.
- **Failure mode:** Autogenerate detected 15+ unrelated changes (dropped indexes, type changes, dropped tables) because the live DB schema had drifted from the model definitions over multiple sessions. The generated migration would have made destructive changes (dropping uart_sessions table, removing indexes).
- **Evidence:** Migration output included `drop_table('uart_sessions')`, `drop_index('idx_findings_source')`, `alter_column` type changes on sbom_components timestamps.
- **How to avoid:** Always review autogenerated migrations before applying. For simple column additions, write the migration by hand — it's faster and safer than cleaning up autogenerate output. Reserve autogenerate for greenfield schemas or after a full schema reconciliation.

### 2. Docker GID mismatch not caught at container build time
- **What was done:** Backend Dockerfile uses `DOCKER_GID` build arg (default 999) to create a docker group. The host Docker socket has GID 145 (Raspberry Pi). The mismatch was only discovered at runtime when emulation failed with `PermissionError(13)`.
- **Failure mode:** Silent failure — container builds successfully, backend starts, but emulation calls fail with an opaque permission error. No healthcheck or startup validation catches the mismatch.
- **Evidence:** `stat -c '%g' /var/run/docker.sock` returns 145, but `.env` had no `DOCKER_GID` set, so docker-compose defaulted to 999.
- **How to avoid:** Document `DOCKER_GID` in `.env.example` with a comment showing how to detect it (`stat -c '%g' /var/run/docker.sock`). Consider adding a startup check in the backend that tests Docker socket access and logs a clear error message if it fails.

### 3. Frontend type-checking unavailable in dev workflow
- **What was done:** Attempted to run TypeScript type-checking to verify changes before building.
- **Failure mode:** No `npx`/`node` on the host (Raspberry Pi), and the production frontend container doesn't include Node.js — only nginx serves the built files. Had to rely on `docker compose build frontend` succeeding as the only type validation.
- **Evidence:** `npx tsc --noEmit` failed on host, `docker exec frontend npx tsc` also failed. Only the Vite build in the Dockerfile caught type errors.
- **How to avoid:** This is acceptable for the Raspberry Pi dev environment — the Docker build is the CI equivalent. For faster feedback, could add a `frontend-dev` compose profile with Node.js available, but the build is fast enough (~20s) that this isn't blocking.
