## Successful Patterns

### 1. Reuse of resolve_firmware dependency across all firmware-scoped pages
- **Description:** The `resolve_firmware()` FastAPI dependency in `deps.py` already accepted an optional `firmware_id` query parameter. Adding firmware selectors to EmulationPage, FuzzingPage, and ComponentMapPage required only frontend changes — adding the FirmwareSelector component and passing `firmware_id` to API functions.
- **Evidence:** 3 pages updated with zero backend router changes. Only the API client functions needed a new optional parameter. Confirmed session 5 pattern #1.
- **Applies when:** Any new page that shows firmware-specific data. Always check `resolve_firmware` first — the backend likely already supports firmware_id selection.

### 2. FirmwareSelector as a drop-in component pattern
- **Description:** The FirmwareSelector component follows a consistent integration pattern: (1) import `listFirmware` + `useProjectStore` + `FirmwareSelector`, (2) add `firmwareList` state + `selectedFirmwareId` from store, (3) fetch firmware list in useEffect, (4) place `<FirmwareSelector>` in header, (5) pass `selectedFirmwareId` to API calls.
- **Evidence:** Same 5-step pattern applied identically to EmulationPage, FuzzingPage, ComponentMapPage, and FindingsPage. Each integration took ~5 minutes.
- **Applies when:** Any page that should support firmware version selection. The component auto-hides when <=1 firmware exists, so it's safe to add everywhere.

### 3. Adding a nullable FK column to enable filtering without breaking existing data
- **Description:** Added `firmware_id` as a nullable FK to the Finding model. This allows filtering findings by firmware version while keeping all existing findings (which have NULL firmware_id) still visible when no filter is applied. The migration is minimal — one column, one index, one FK constraint.
- **Evidence:** `81f49fd099f5` migration adds the column. Existing findings remain accessible. Security audit and YARA scan now tag new findings with the firmware they came from. The list endpoint filters only when firmware_id is explicitly provided.
- **Applies when:** Adding a new dimension to filter existing data. Use nullable FK + "filter only when provided" rather than requiring backfill of historical records.

### 4. Batch-tagging findings by firmware during scan loops
- **Description:** Security audit and YARA scan iterate over firmware_list. Changed the accumulator from `list[SecurityFinding]` to `list[tuple[SecurityFinding, uuid.UUID]]` to track which firmware each finding came from, then pass firmware_id to the persist function.
- **Evidence:** `security_audit.py` — both scan loops now carry firmware.id alongside each finding. Simple tuple change with no structural refactoring.
- **Applies when:** Any scan that iterates over multiple firmware versions and creates records. Always track which firmware produced each result at the loop level.

### 5. Navigation state for cross-page linking
- **Description:** Used React Router's `location.state` to pass a `findingId` when navigating from SecurityScanPage to FindingsPage, allowing auto-selection of the clicked finding. Combined with `?path=` query params for file navigation to ExplorePage.
- **Evidence:** SecurityScanPage finding titles use `<Link state={{ findingId }}>`, FindingsPage reads it via `useLocation()`. File paths use `?path=` which ExplorePage already handled.
- **Applies when:** Cross-page navigation where the target page should auto-select or scroll to a specific item. Use `state` for IDs (not in URL), `?param=` for paths that the target page already handles.

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Nullable firmware_id on findings (not required) | Existing findings have no firmware association; backfill is impractical since scan source isn't recorded | Correct — zero data migration needed, filtering works for new findings |
| Docker GID fix as manual .env step | .env is gitignored and hook-protected; varies per host | Correct — GID 145 on this Raspberry Pi host, default 999 on most systems |
| Alembic migration hand-written (not autogenerated) | Autogenerate picked up 15+ unrelated index/type drift changes | Correct — clean 3-operation migration vs. noisy auto-generated one |
| FindingsList file_path NOT made clickable | File path is inside a button (whole row is clickable); adding a nested link is bad UX | Correct — FindingDetail already provides the clickable file link |
| Findings diff deferred to future session | Moderate complexity feature that needs design thought (how to match findings between versions) | Appropriate — firmware_id foundation is in place, diff can build on it |
