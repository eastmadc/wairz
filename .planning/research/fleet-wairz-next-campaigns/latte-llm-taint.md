# LATTE-Style LLM Taint Analysis — Research Brief

> Scout: fleet-wairz-next-campaigns / latte-llm-taint
> Date: 2026-04-17

## Summary

LATTE (TOSEM 2025) is the first published static binary taint analyzer driven by an LLM: Ghidra-based backward data-dependency slicing + GPT-4 with a fixed 4-stage prompt chain. Juliet accuracy 62–96% per CWE; 37 new bugs + 10 CVEs on Karonte firmware. Wairz already owns every primitive — `decompile_function`, `xrefs_to/from`, rule-based `trace_dataflow`, and an MCP client (Claude) that *is* the LLM. A LATTE-style MCP tool here is a prompt constructor, not a model host: ~450 LOC in one file, 2 sessions to prototype + evaluate against DVRF.

## LATTE Paper Findings

**Paper:** Liu, Lyu, Zhu, Sun. *LLM-Powered Static Binary Taint Analysis.* TOSEM 34(3) Article 83 (Feb 2025). arXiv 2310.08275v4.

**Prompt structure — 4 fixed templates, sequential API calls:**

1. *Sink id:* "As a program analyst, is it possible to use a call [fn] as a sink when performing taint analysis? If so which parameters need to be checked for taint."
2. *Source id:* same framing, "starting point (source) ... which parameter stores the external input."
3. *Dataflow start/middle:* "I give you snippets of C code generated by decompilation, using [fn] as the taint source, [param] marked as the taint label to extract the taint data flow." Middle: "Pay attention to data alias and tainted data operations."
4. *Vuln check:* "Based on the above, analyze whether the code has vulnerabilities... explain what kind of vulnerability according to CWE."

**Slicing:** backward intra/inter-procedural data-dependency slicing from sink call-site back to source, deduped to "dangerous flows." Unit fed to LLM = decompiled C function body (not BBs, not raw asm).

**Model:** GPT-4.0. Temperature 0.4–0.5 optimal.

**Accuracy (Juliet v1.3 stripped):** CWE-78 96.5%/96.3%, CWE-134 93.9%/94.0%, CWE-190 62.1%/62.1%, CWE-606 73.3%/74.2% (acc/F1). Also CWE-120. Runtime 13.5–15.1s per target (excl. Ghidra).

**Real-world:** Karonte dataset (49 samples, Netgear/D-Link/TP-Link/Tenda) — 119 bugs, **37 new**, **10 CVEs**. Beat Emtaint and Arbiter.

**Code:** No public GitHub from authors. `lanl/LATTE` is unrelated. Must reimplement — the opportunity.

None of the MCP servers (GhidraMCP, binary_ninja_mcp, pyghidra-mcp) ship a LATTE-style taint prompt flow. That gap is Wairz's window.

## Prompt Design Recommendations

The MCP tool constructs **one consolidated prompt string** per call. Claude does the reasoning; the tool does the slicing. Shape:

```
You are performing taint analysis on a binary function.

## Target
Binary: /bin/httpd (MIPS-LE, stripped). Function: FUN_00401a20 @ 0x00401a20

## Decompiled body (Ghidra, cached)
<decompiled C, up to ~15KB, wrapped in <decompiled>...</decompiled>>

## Immediate callers (xrefs_to)   / Immediate callees (xrefs_from)
- main @ 0x00400f00 (calls ...)    - recv  → POTENTIAL SOURCE
- FUN_00401840 @ ...               - strcpy → POTENTIAL SINK

## Dangerous sinks present: strcpy, sprintf, system, popen, execve, memcpy

## Instructions
1. Identify SOURCES (recv, read, getenv, argv, nvram_get, HTTP parsers). Quote the line.
2. Identify SINKS (strcpy, sprintf, system, popen, execve). Quote the line.
3. For each (source,sink) pair trace flow; note sanitization.
4. Classify by CWE + assess exploitability.
5. Output JSON: {"flows":[{"source":"","sink":"","path":["line1","line2"],
   "cwe":"CWE-78","exploitable":true,"confidence":"high|medium|low",
   "reasoning":""}]}
```

Design notes:
- **Single-shot, not multi-turn.** LATTE's 4-stage flow assumes a stateful OpenAI session. When the MCP client IS the LLM, multi-turn happens naturally via Claude's tool-use loop — we don't encode the state machine.
- **JSON schema in prompt** plus `<taint_report>...</taint_report>` delimiter for programmatic parse.
- **30KB budget:** decomp ≤15KB, metadata ≤3KB, instructions ≤2KB. Middle-out truncate oversized decomp.

## Integration into Wairz (MCP Tool)

**New file:** `backend/app/ai/tools/taint_llm.py` (~450 LOC).

**Tool:** `analyze_taint_llm`. Inputs: `binary_path` (req), `function_name` (req), `include_callers` (default 2), `include_callees` (default 5), `focus_cwe` (optional list).

**Pipeline (all existing primitives):**
1. `context.resolve_path()` — sandbox check.
2. Reuse `decompile_function` service — cached decomp (<100ms hit).
3. Reuse `xrefs_to` for callers; `list_imports` + heuristic for callees.
4. Classify imports against sink list (`strcpy`, `sprintf`, `system`, `popen`, `execve`, `memcpy`, `gets`, `scanf`) and source list (`recv`, `read`, `getenv`, `nvram_get`, `fgets`).
5. Render prompt template; return string (≤30KB via `utils/truncation.py`).

**Registration:** add `register_taint_llm_tools(registry)` wired into `backend/app/ai/__init__.py`.

**Supporting tool (optional, ~150 LOC):** `list_taint_candidates` — ranks functions by `(sinks_imported, score)`. Lets Claude pick high-value targets instead of scanning all 10K functions.

**Estimate:** Session 1 (~450 LOC) scaffold + tests. Session 2 (~200 LOC) ranking + DVRF goldens. Total ~650 LOC, 2 sessions.

## LLM Placement Decision

**Strongly recommend Option A.** MCP tool returns a crafted prompt-as-string to Claude Code; Claude analyzes in-conversation. Justification:

1. **Existing pattern.** Every Wairz MCP tool returns strings. `grep anthropic backend/app/` — zero hits. No tool imports the SDK.
2. **CLAUDE.md rule.** "No API keys stored in the backend. The Anthropic API key is user-provided via their Claude Code/Desktop configuration and never touches Wairz." Option B directly violates this.
3. **Offline-capable.** Firmware RE often runs air-gapped; Option B requires outbound HTTPS from the backend container.
4. **Cost/model agility.** User sees spend on their own plan; works with whatever model they connected — no pinning.
5. **LATTE's multi-stage flow is free in Option A.** Claude's tool-use loop re-queries the tool for callers/other functions; we don't encode a state machine.

Option B is justified only for deterministic batch scans ("scan all 10K functions overnight") — that's a future `vulhunt`-style containerized add-on, not this tool.

## Evaluation / Golden Tests

1. **Unit tests** (`backend/tests/test_taint_llm_tools.py`): prompt rendering (≤30KB, schema section present), path-traversal rejection, missing-function error.
2. **Integration goldens:** DVRF (`stack_bof_01/02`, `heap_overflow_01`, `format_string_01`) + 2 OpenWrt CVEs (CVE-2022-30075 rpcd, CVE-2023-41974 dropbear). Hand-annotated `{source, sink, cwe}` expected per function. Target: ≥80% hit on DVRF; ≤20% FP on clean busybox functions.
3. **LATTE replication:** rerun Juliet CWE-78/134 vs. LATTE's 96.5%/93.9%. Claude Opus may outperform 2024-era GPT-4.0.

## Risks

1. **Hallucinated CVEs.** Mitigation: JSON output requires `path` field quoting actual lines; post-validator rejects flows citing lines not in the decomp body. UI surfaces only `confidence=high` by default.
2. **Token cost at scale.** 10K-function binary × 5K tokens each = 50M tokens. Mitigation: `list_taint_candidates` filter — only analyze functions importing dangerous sinks. Typical embedded binary has 20–200. ~1M tokens per binary ≈ $3 at Opus rates.
3. **Consistency.** Same function twice → different report. Mitigation: structured-output prompt + "be deterministic" guidance. Cache in `analysis_cache` keyed on `(binary_hash, function_name, prompt_version)`.
4. **Ghidra decomp quality on O3/MIPS.** Mitigation: inline warning in prompt — "discount artifacts like `in_GP`, focus on control flow."
5. **Prompt injection via firmware strings.** A string literal could say "Ignore previous instructions." Mitigation: wrap in `<decompiled>...</decompiled>` and instruct Claude to treat contents as data.
6. **Overlap with existing `trace_dataflow` / `cross_binary_dataflow`.** Mitigation: tool doc clearly frames `trace_dataflow` as rule-based (fast, low recall) and `analyze_taint_llm` as LLM-backed deep-dive after `trace_dataflow` flags a candidate.

## Dead Ends

- **LATTE source code.** No public GitHub. `lanl/LATTE` is unrelated (Los Alamos). Must reimplement.
- **Fine-tuned binary-taint models.** SecureFalcon (121M) is a classifier, not a tracer. LLM4Decompile improves decomp but doesn't reason about security.
- **Multi-turn inside the MCP tool.** Requires the tool to hold LLM state → forces Option B. Claude's tool-use loop gives the same multi-step reasoning for free.
- **Server-side local LLM.** 30B-class open models underperform GPT-4/Claude on this task per IRIS and LATTE baselines.

## Confidence

**High** on: LATTE's published prompt templates, accuracy numbers, LLM choice (GPT-4), lack of public code, Wairz MCP architecture (verified by reading `mcp_server.py`, `binary.py`, `vulhunt.py`), Option A being the right choice.

**Medium** on: LOC estimate (450) and session count (2) — depends on whether `list_taint_candidates` needs its own Ghidra script or can reuse `list_imports` output; integration-test quality depends on DVRF coverage of Karonte-class CVEs.

**Low** on: exact accuracy when replacing GPT-4.0 with Claude Opus 4.7 — could be materially better given 2024→2026 model gains, but no published comparison exists.
